# M20Pro 跨楼层巡检导航

## 当前链路

系统只保留一套统一导航。单层任务是只有一个楼层段的统一任务，跨层任务只是在相邻楼层段之间多一条有向楼梯连接边。

```text
统一任务
-> stair_executor
-> Nav2 到楼梯入口
-> /m20pro/gait_command 切换楼梯步态
-> /cmd_vel_nav -> command_mux -> 103 运动链
-> 到达 source_platform 并停车
-> 并行激活 104/106 预置目标地图并执行 2101 重定位
-> Nav2 离开楼梯
-> 恢复 flat 步态
-> 继续原任务
```

上楼使用正向低速，下楼按 X30Pro 原厂操作要求使用后退低速。速度仍经过现有 `command_mux`，不新增绕过仲裁直发 103 的通道。

旧 `floor_manager.use_stairs()` 只保留明确拒绝，防止历史入口与新链并行。已删除第二个动作编排节点。`terrain_guard` 当前只可在 106 做影子观测，不作为首轮跨层运动的前置门禁，也不改变平地 `/scan`。

`floor_manager` 只负责同楼层 Nav2 目标和楼层上下文，不再根据地图中的 terrain segment 或收到普通 Nav2 目标自动切换步态。`stair_executor` 在入口导航前明确使用 `flat`，到达入口后切换楼梯步态，越过目标层出口后再恢复 `flat`，避免两套机制同时控制机器狗。

## 成功定义

一条跨层任务只有在以下顺序全部完成后才算成功：

1. Nav2 到达源层楼梯入口。
2. 机器狗切换正确楼梯步态并到达共享平台。
3. 104 Nav2 与 106 原厂目标地图已并行激活。
4. 目标层 `target_platform` 的 2101 重定位成功。
5. Nav2 到达目标层出口，恢复平地步态并继续任务点。

仅前端楼层文字发生变化不算切层成功。

## 建图与路线

1. 在前端建图页填写项目、建筑/区域、地图名称和现场真实楼层。楼层不限于 `F19/F20/F21`。
2. 每层分别执行“启动 106 建图”、“完成/保存建图”和“拉取 106 建图结果”。拉取只归档，不代替地图切换和重定位。
3. 在两层正式地图上分别完成重定位，然后标定一条有向路线的四个点。

| 点位 | 所属地图 | 用途 |
| --- | --- | --- |
| `entry` | 源层 | Nav2 结束平地导航、准备切步态 |
| `source_platform` | 源层 | 楼梯直行的停车目标与切图位置 |
| `target_platform` | 目标层 | 切图后 2101 重定位位姿 |
| `post_exit` | 目标层 | Nav2 离开楼梯后恢复平地步态的目标 |

`source_platform` 和 `target_platform` 描述同一个物理共享平台在两张地图中的坐标。该平台必须是两张定位图都具有稳定特征的重叠停车区，并在前后留出停车余量；不能把切换点画在地图边界线上。上楼和下楼是两条有向路线，必须分别实测和保存，不由系统自动镜像生成。

任务开始前，目标层 Nav2 地图必须已归档在 104，目标层原厂地图包必须已存在于 106。运行时只做激活，不下载、复制或重新生成地图；机器狗停车后同时发起 104 `load_map` 和 106 `drmap apply`，两端成功后在 `target_platform` 自动执行 2101。原厂定位器一次只能激活一张地图，因此停车切换不可消除；首次现场测试应记录两端激活和 2101 的真实耗时，再确定可达到的暂停时间。

## 任务执行

1. 选择首个任务点所在的地图并完成重定位。
2. 在任务页按真实顺序加入任务点。点位顺序就是执行顺序，系统会在连续楼层段之间解析已保存的有向路线。
3. 确认路线预览没有“缺少路线”，然后生成并执行统一任务。
4. 真实运动必须使用 `104_start_real_move.sh`。`104_start_real_shadow.sh` 不启动楼梯执行器，不用于跨层运动验收。

任务可以重复执行。每次启动都会根据当前点位顺序和当前有向路线刷新统一计划，不把上一次运行保存的计划当成路线认证。连接边开始后直到目标层出口 Nav2 完成，任务主循环都不会并发下发后续任务点；目标楼层文字先更新也不会提前释放执行权。

## 首轮必要判定

为先跑通最小闭环，起步前只检查：

- 有向路线具有路线、源/目标地图和源/目标楼层身份；
- 方向是 `up` 或 `down`；
- `entry/source_platform/target_platform/post_exit` 四个位姿完整；
- Web 直接向唯一 `stair_executor` 发布连接边请求，不用心跳 freshness 作为前置许可证；执行器收到请求后只核对自身确实已启用且没有执行上一条路线。

运动中只保留直接影响闭环的停止条件：

- 用户停止任务；
- 机器狗地图位姿超过 `motion_pose_timeout_s` 未更新（通信超时）；
- 当前阶段超过 `connector_stage_timeout_s`；
- 入口或出口 Nav2 明确返回不可用、拒绝或失败；
- 104/106 切图或 2101 重定位失败。

首轮不要求 terrain 状态、走廊认证、执行器心跳 freshness、另一个动作编排器或多层运动 readiness 同时通过。请求发出后若执行器没有任何状态回执，现有通信超时会停止任务。后续只根据真机实测出现的明确风险增加加固项。

## 现场参数

只在 `src/m20pro_bringup/config/m20pro_field_profile.yaml` 修改现场参数。楼梯首次测试的核心项为：

```yaml
stair_transition:
  motion_speed_mps: 0.12
  motion_command_hz: 10.0
  motion_pose_timeout_s: 1.50
  gait_settle_s: 1.0
  connector_stage_timeout_s: 180.0
  platform_position_tolerance_m: 0.50
```

修改后执行：

```bash
./scripts/apply_field_profile.sh --check
./scripts/apply_field_profile.sh
```

真机部署前可在上位机验收 ROS 适配层：

```bash
source /opt/ros/humble/setup.bash
source install/setup.bash
python3 scripts/smoke_stair_executor_ros.py
python3 scripts/smoke_stair_floor_manager_ros.py
```

两个脚本都强制使用仅本机 DDS、独立 ROS domain 和私有话题，不会向机器狗默认 domain 发送速度。第一个验证执行器自身的上/下楼状态顺序；第二个同时启动安装态 `floor_manager` 和 `stair_executor`，用本机假 Nav2 action server 验证入口/出口目标及 `flat -> stair_up/down -> flat` 的真实步态时序，并在共享平台直接调用生产 Web 的切图事务方法，以确定性适配器代替真实 104/106 和 2101，验证请求身份、目标楼层同步和切图回执能继续推进出口 Nav2。

## 最小现场验收

1. 只选择一段规则、空间充足、中间有可停车平台的固定楼梯。
2. 只配置两层、一条上行路线、每层一个近距离任务点。
3. 先分别验证两张地图的人工切换和重定位，再执行上行单程。
4. 现场人员在机器狗侧后方看护，原厂手柄急停随时可用，全程录包。
5. 上行完整通过后，再单独配置和验证反向下楼路线。
6. 首次验收不同时叠加雷达、YOLO、长距离多点和复杂巡检。

当前分支已完成离线合同测试、安装节点的 ROS 话题级上/下楼回放和启动接线，但尚未在真实楼梯完成首轮运动验收，因此不应宣称已可无人跨层运行。
