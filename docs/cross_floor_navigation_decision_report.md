# M20Pro 跨楼层导航技术路线决策报告

**决策日期：** 2026-07-24

**决策状态：** 已进入最小真机闭环实现

**适用范围：** M20Pro 单层与跨楼层巡检导航

## 决策结论

M20Pro 采用一套统一导航系统：楼层内继续使用已验证的 Nav2 和 106 `/scan`，楼梯段由单一 `stair_executor` 执行，共享平台继续复用现有 104/106 切图事务和 2101 重定位。

这不是在现有系统旁边新建一套“跨层导航”。单层是统一任务只包含一个楼层段的情况，跨层只在相邻楼层段之间增加一条有向楼梯连接边。

当前开发优先级明确为：

> 先用唯一、最小的运动链跑通真实跨层闭环，再根据实测失效模式逐项加固。

## 为什么就用这套方案

### 保留已经跑通的平地主链

当前单层导航的核心是：

```text
106 原厂 drmap 定位 -> 104 tcp_bridge -> 地图位姿/TF
106 /LIDAR/POINTS -> 106 edge scan -> /scan -> 104 Nav2 代价地图
```

楼梯功能不更换定位源，不替换 `/scan`，不恢复旧扫描选择器，也不把原始点云重新跨主机传输。这从架构上隔离了楼梯开发对平地避障的影响。

### 符合四足机器人的真实运动边界

Nav2 擅长楼层平面内的自由空间规划，但楼梯段需要原厂楼梯步态和确定运动方向。因此 Nav2 只负责到达入口和离开出口；楼梯本体由执行器低速直行，上楼向前，下楼按原厂要求后退。

这样做不需要在楼梯上调教 DWB 自由规划，也不会为楼梯新建一套平地导航。

### 直接复用现有切图和重定位能力

当前工程已经具备：

- 现场真实楼层和每层地图身份；
- 104 Nav2 地图与 106 原厂地图包；
- 有向楼梯路线和四个关键位姿；
- 104/106 切图事务；
- 目标层平台 2101 自动重定位。

新链只把“机器狗怎么从入口到共享平台”闭合起来，不重写地图系统和定位系统。

## 唯一运行链

```text
统一任务执行器
  -> 解析当前有向楼梯路线
  -> stair_executor 导航至 entry
  -> 切换 stair_up / stair_down
  -> 经 /cmd_vel_nav 和 command_mux 到 source_platform
  -> 在共享重叠平台零速并行激活 104/106 预置地图
  -> 在 target_platform 执行 2101
  -> Nav2 导航至 post_exit
  -> 恢复 flat 步态
  -> 统一任务继续下一点
```

已删除 `stair_action_orchestrator`。旧 `floor_manager.use_stairs()` 只保留拒绝，防止旧链被误调用；它不参与新跨层任务。楼梯速度进入现有 `command_mux`，不新增第二套仲裁。

切图不在地图边界线上触发，而是在两张定位图都有稳定特征、且留有停车余量的共享重叠平台触发。任务前必须把 104 Nav2 地图和 106 原厂地图包全部准备好；运行时并行执行 104 `load_map` 与 106 `drmap apply`，随后在目标图平台坐标自动执行 2101，不把下载、哈希计算或地图生成放进停车窗口。原厂定位器一次只能激活一张图，因此零停顿不现实；暂停时间以首次真机对两端激活和 2101 的实测为准。

## 首轮为什么不启用 terrain 硬门禁

106 本地三维地形感知从长期方向上有价值，但它尚未在 M20Pro 真实楼梯上完成误报、漏报和时效性标定。在最小运动闭环前把它变成强制条件，只会把未定标的感知状态变成新的运行故障源。

因此首轮实现中：

- `terrain_guard` 只允许在 106 影子运行；
- 它不修改或选择 `/scan`；
- 它不阻止楼梯路线启动和平台切图；
- 先从录包中证明它能可靠识别人员、箱体、踏面缺口和点云中断，再接入停止逻辑。

## 首轮必要判定

路线启动前只检查路线身份、方向、四个必要位姿和唯一执行器是否空闲。运动中只保留以下直接停止条件：

- 用户停止任务；
- 地图位姿通信超时；
- 阶段超时；
- 104/106 切图或 2101 重定位失败。

这些判定不是额外认证体系，而是防止在已经无法闭合的情况下继续发送速度。

## 性能影响

首轮方案的增量开销很小：

- 楼梯执行器是低频状态机，楼梯段以 10 Hz 发布速度；
- 不跨主机传输原始点云；
- 不引入新定位器、新全局规划器或新代价地图；
- `shadow` 模式不启动楼梯运动节点；
- 平地任务的数据路径和算法开销不变。

因此这套方案不会因为支持跨层就持续拖慢单层导航。

## 现场验收顺序

1. 两层、一条上行路线、每层一个近距离任务点。
2. 先手动验证两张地图切换和重定位，再跑一次上行完整链。
3. 上行通过后再配置反向路线测下楼。
4. 往返通过后再加多点、停留、雷达和 YOLO。
5. 每个阶段都录包，只对已观测的失效模式加固。

当前实现已完成离线合同、启动接线和状态顺序验证，但未经真实楼梯首轮运动验收。这个阶段的正确结论是“已具备可测试的最小闭环”，不是“已可无人跨层运行”。

## 明确不采用

- 不整体迁移 DDDMR、Open-RMF 或其他导航栈。
- 不复制 X30Pro 闭源程序。
- 不让普通 DWB 在楼梯上自由绕行。
- 不恢复会影响平地避障的楼梯扫描选择链。
- 不在 104 长期接收 106 原始点云。
- 不为跨层建第二个任务队列、第二个动作编排节点或第二套仲裁。
- 不在首轮真机闭环前叠加尚未定标的 terrain 门禁和路线认证。
