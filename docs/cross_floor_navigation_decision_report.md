# M20Pro 跨楼层导航技术路线决策报告

**决策日期：** 2026-07-22
**决策状态：** 正式技术路线
**适用范围：** M20Pro 跨楼层建图、楼梯通行、地图切换和巡检任务执行

## 一、决策结论

M20Pro 跨楼层能力正式采用以下唯一技术路线：

> **以现有 M20Pro 主线为主体，建立“建筑语义路线图 + 楼层内 Nav2 + 专用楼梯执行器 + 106 三维地形安全感知 + 104/106 原子切图”的分层跨楼层系统。**

这不是引入另一套导航系统，也不是把若干开源包拼接到现有工程中。现有单层导航、任务、地图、前端、103 TCP 桥和 104/106 切图事务继续作为正式主链；开源项目只提供经过验证的设计思想和算法依据。

该路线的核心原则为：

1. 平地继续由当前 Nav2 和 `/scan` 负责。
2. 楼梯只允许由专用执行器通过显式路线执行，禁止普通 DWB 在楼梯上自由规划。
3. 106 原始点云只在 106 本地用于三维地形判断，不重新跨主机传输原始点云。
4. 三维地形感知只给楼梯执行器提供通行性和危险状态，不替换、不裁剪、不选择 Nav2 的 `/scan`。
5. 楼层切换继续使用现有 104/106 原子事务和 2101 重定位确认。
6. 任一证据缺失、过期或相互矛盾时立即停止，不能猜测通过。

## 二、为什么这条路线最稳妥

### 2.1 它保护已经跑通的单层导航

当前经过现场验证的基础链由定位和障碍感知两个并行出口组成：

```text
106 原厂 drmap 定位 -> 104 TCP bridge -> map 位姿与 TF
106 /LIDAR/POINTS -> 106 edge scan -> /scan -> 104 Nav2 代价地图/前端/自检/录包
```

实机主线不启动 AMCL，原厂定位和 TCP bridge 是唯一定位权威。楼梯能力不得改变上述链路的消息类型、话题、QoS、障碍高度带或代价地图入口。这样即使楼梯模块未启动、异常退出或尚未完成，单层定位、导航和动态避障仍按原链运行。

旧方案的问题不是少调了几个参数，而是把平地障碍解释和楼梯障碍解释放进同一个扫描出口，导致平地避障能力受楼梯逻辑影响。新路线从结构上取消这种耦合，因此不会再次出现“为了能看见台阶而削弱正常障碍物”的回归。

### 2.2 它符合四足机器人的真实控制边界

Nav2 擅长二维平面内的自由空间规划，但楼梯不是普通二维可通行区域。楼梯段需要同时满足：

- 使用正确的本体步态；
- 保持入口朝向和纵向路线；
- 上下楼采用不同方向；
- 约束横向偏移和转向；
- 持续判断踏面、楼梯边界和新增障碍；
- 在异常时原地停止，而不是执行旋转、倒退或绕行恢复。

因此楼梯段必须由确定性执行器负责。Nav2 只负责把机器狗送到安全入口，并在机器狗离开楼梯区域后继续楼层内导航。

### 2.3 它复用现有最可靠的切层资产

当前系统已经具备以下可继续使用的能力：

- 现场真实楼层和地图身份；
- 每层 104 Nav2 地图和 106 原厂地图包；
- 有向跨楼层路线数据模型；
- 两侧共享平台坐标字段；
- 104/106 协调切图骨架；
- 目标层 2101 重定位接口；
- 切图失败补偿和状态不确定保护的实现骨架；
- 跨楼层任务点顺序和前端流程展示。

这些是可以复用的基础，不等于跨层事务已经达到无人运行标准；当前代码缺少的成功门禁、回滚闭环和崩溃恢复必须先补齐。新路线只补上“从楼梯入口安全到达共享平台”以及这些事务缺口，避免第三套地图、任务或楼层机制进入工程。

### 2.4 它避免依赖不可控的外部运行环境

正式系统不依赖完整 Open-RMF、CUDA、GPU 版高程图、ROS1 运行时或 X30Pro 闭源二进制。三维地形算法应以可在 106 上运行的轻量 C++ 实现为准，输入复用 106 已经可读的原始点云，输出为固定结构的轻量状态。

这使部署、排错和性能边界仍掌握在当前三主机架构内，不引入额外服务器或庞大运行栈。

## 三、最终系统架构

```text
                         前端跨楼层巡检任务
                                  |
                         建筑语义路线图规划
                                  |
            +---------------------+---------------------+
            |                                           |
       楼层内导航边                                  楼梯连接边
            |                                           |
      104 Nav2 + /scan                          专用楼梯执行器（104）
            |                                           |
   正常规划、动态避障                         步态/方向/速度/进度状态机
                                                        |
                                            106 三维地形安全状态
                                                        |
                                           106 原始 PointCloud2
                                                        |
                              到达共享平台并确认楼梯段结束
                                                        |
                         104/106 原子切图 + 2101 重定位
                                                        |
                                      目标层 Nav2 继续巡检
```

系统只保留两类运动边：

- **楼层内导航边：** 由 Nav2 执行，允许正常规划和动态绕障。
- **楼梯连接边：** 由专用楼梯执行器执行，只允许沿已标定走廊通过，遇障停止，不在台阶上绕行。

## 四、建筑语义路线图

跨楼层任务不能再理解为“一串带楼层字符串的点”。正式模型应包含建筑、楼层、地图和连接器四层身份：

```text
site
  -> building
     -> floor
        -> map version
        -> task points
        -> stair connectors
```

每条楼梯连接边至少保存：

- 唯一路线 ID；
- 起始楼层和目标楼层；
- 起始地图 ID、目标地图 ID 及版本；
- 入口点、起始层平台点、目标层平台点、出口点；
- 上楼或下楼方向；
- 前进或后退要求；
- 楼梯步态；
- 允许速度和入口距离；
- 对应三维楼梯走廊版本；
- 感知配置版本；
- 失败处理策略。

路线必须是有向的。`F1 -> F2` 和 `F2 -> F1` 是两条不同的边，不能通过交换起终点自动生成。下楼路线必须显式标记后退方向，不能由前端或执行时临时推断。

该模型借鉴 Nav2 Route Server 的有向语义边思想，但直接落在现有 M20Pro 路线合同中，不把新版本 Nav2 Route Server 强行移植到 Foxy。

## 五、建图和地图生成方案

### 5.1 正式建图产物是楼层地图集

当前 M20Pro 前端的多楼层会话会按楼层分别调用 `drmap mapping`，每层生成一个独立的 106 原厂地图包。它不是 X30Pro 那种一张连续的整栋楼三维地图，正式方案不得把两者混为一谈。

M20Pro 的一个建筑项目由以下资产共同构成：

```text
building map set
  -> floor maps[]
  -> stair corridors[]
  -> directed stair routes[]
  -> shared-platform anchors[]
```

三类地图资产职责必须分开，不能用一张二维图同时承担定位、平地规划和楼梯几何判断：

| 资产 | 覆盖范围 | 运行职责 |
| --- | --- | --- |
| 106 原厂定位图 | 本层作业区；作为某条有向边的源图时，还必须保持定位到该边的切换平台 | 原厂定位、2101 重定位 |
| 104 二维占据图 | 本层平地、楼梯入口和出口；楼梯本体保持非 Nav2 通行区 | 平地规划、动态避障 |
| 三维楼梯走廊 | 入口、完整台阶、边界、平台和出口 | `terrain_guard` 判断与 `stair_executor` 约束 |

对每条有向楼梯边，源、目标两张 106 定位图必须在该边的物理切换平台形成可观测重叠区，但允许拥有不同的地图坐标系。反向路线是另一条边，必须单独确认其切换平台和两张图的覆盖；不能假定正向路线反过来就成立。

同一物理平台必须保存两套锚点：

- `source_platform`：平台在起始层地图坐标系中的 `x/y/z/yaw`；
- `target_platform`：平台在目标层地图坐标系中的 `x/y/z/yaw`。

切图时机器狗保持在同一物理停车区和朝向，只把定位坐标从 `source_platform` 转换为 `target_platform`。系统不使用楼层编号、地图原点或楼高估算这个转换。

每张 104 地图、106 原厂包和三维走廊都必须保存不可变内容摘要。路线绑定明确的地图 ID、版本和摘要；只比较地图宽高、分辨率和原点，不足以证明加载的是同一张地图。

### 5.2 逐层建图流程

一个相邻楼层项目按以下固定流程建立：

1. 在前端创建一个包含实际楼层顺序的多楼层项目会话。
2. 建起始层地图，扫描本层作业区、楼梯入口，并由人工遥控使用原厂楼梯步态通过楼梯，继续扫描到本方向的切换平台及平台周边结构。
3. 保存并拉取起始层地图；同时由 106 本地记录本次楼梯段点云，生成起始方向的走廊数据，不实时跨主机传点云。
4. 在目标层新建独立地图，从同一切换平台开始，充分扫描平台周边和目标层作业区。需要支持反向路线时，必须按反向边的切换位置再次验证两张 106 地图的覆盖，并人工采集反向走廊数据。
5. 保存并拉取目标层地图，在两张地图中分别重定位到同一物理平台，记录 `source_platform` 和 `target_platform`。
6. 系统依次验证两张地图在平台处均能稳定定位、`/scan` 新鲜、代价地图可更新，并锁定三类资产摘要；验证通过后才允许保存有向楼梯路线。

建图阶段的楼梯通过必须人工遥控并有人看护，不能使用尚未验收的自主楼梯执行器。

### 5.3 共享平台锚点必须标定和验收

共享平台不是在地图上随手标两个点。现场必须设置明确的停车区和朝向基准，以同一个本体参考点完成标定：

1. 人工从不同初始位置多次进入停车区，确认零速、平地姿态和朝向均可重复。
2. 分别在源图和目标图中执行多次独立重定位，记录 `x/y/z/yaw` 样本、散布和定位稳定窗口。
3. 用停车区安全边界和实测散布确定位置、朝向容差；容差进入唯一现场配置，不写死在代码中。
4. 锚点绑定两张地图的内容摘要和有向路线 ID，任一地图变化后自动失效并要求重新标定。
5. 平台缺少可识别结构、重复定位不收敛或停车误差超过安全边界时，该路线不能发布。

运行时只有机器狗确实进入该停车区并满足源锚点位置、朝向、零速和姿态门限，才允许使用固定 `target_platform`。这样切图不会凭空引入机器狗实际停车误差。

### 5.4 原始地图永不覆盖

每层建图完成后保留两类资产：

1. **原始地图：** 106 原厂建图结果及拉回 104 的原始栅格图，作为可追溯真值。
2. **派生导航图：** 根据原始地图、三维点云和已确认楼梯走廊生成，只供导航使用。

前端地图修饰、自动楼梯走廊生成和现场校准均生成新版本，不直接覆盖原图。任务和路线必须绑定明确的地图版本，避免地图修改后旧路线继续静默运行。

### 5.5 楼梯通道不再靠人工擦白

正式流程为：

1. 操作员在前端标记楼梯大致入口和出口。
2. 离线处理该区域的三维点云，生成高程、坡度、台阶连续性和边界结果。
3. 系统生成候选楼梯走廊和派生二维导航区域。
4. 前端叠加展示原图、候选走廊和风险边界。
5. 操作员只做确认或拒绝，不再使用画笔手工猜测整个楼梯形状。
6. 确认后保存走廊版本，并与有向楼梯连接边绑定。

人工确认不是技术退让，而是安全发布门。算法负责测量和生成，人员只确认结果与现场是否一致。

楼梯本体不需要在二维地图中擦成白色。Nav2 只需要能够到达入口和目标层出口；从入口到共享平台的运动完全由 `stair_executor` 和三维楼梯走廊负责。这样不会为了让二维规划器穿过楼梯而篡改障碍地图。

## 六、106 三维地形安全感知

106 新增唯一的三维地形处理执行体，命名为 `m20pro_terrain_guard`。它只在楼梯任务进入准备阶段后激活，平时不占用持续计算资源。

输入：

- 106 原始 `PointCloud2`；
- 当前本体位姿和姿态；
- 当前楼梯走廊；
- 当前路线方向；
- 对应感知配置版本。

处理结果不能只依赖一个“单级高度上限”，而应至少综合：

- 局部高程；
- 表面法向和坡度；
- 相邻踏面高度变化；
- 踏面连续长度和宽度；
- 左右边界；
- 走廊内悬空、缺口和高物体；
- 点云覆盖率和时效；
- 机器狗俯仰、横滚和本体姿态变化。

输出只允许是轻量结构化状态，例如：

```text
route_id / corridor_version / sequence / timestamp
state = ready | traversable | blocked | unknown | stale
progress / lateral_error / heading_error
hazard_type / hazard_distance / confidence
```

它不得发布替代扫描，不得修改 `/scan`，不得直接控制机器狗。运动权限始终由 104 的楼梯执行器和现有仲裁链掌握。

## 七、专用楼梯执行器

104 在现有 `m20pro_navigation` 包内新增单职责 `stair_executor` 节点，不新建另一套导航 package。`floor_manager` 只负责建筑语义路线编排、楼层状态和切图事务；`stair_executor` 只负责一条已选楼梯边的运动闭环。两者通过版本化的路线请求、执行反馈和最终结果通信，不共享隐式可变状态。

正式状态机如下：

```text
IDLE
  -> NAVIGATE_TO_ENTRY
  -> ALIGN_AT_ENTRY
  -> VERIFY_TERRAIN
  -> SET_STAIR_GAIT
  -> TRAVERSE_STAIR
  -> VERIFY_SHARED_PLATFORM
  -> SWITCH_MAP_TRANSACTION
  -> RELOCALIZE_TARGET_FLOOR
  -> EXIT_STAIR_ZONE
  -> RESTORE_FLAT_GAIT
  -> CONTINUE_MISSION
```

执行职责：

- 入口前仍使用 Nav2；
- 到达入口后取消普通 Nav2 控制权；
- 通过现有仲裁链申请楼梯执行权；
- 调用原厂楼梯步态，按标定方向进行受约束纵向跟踪；
- 根据位姿和三维地形状态限制速度、航向和横向误差；
- 到达共享平台后停止并确认；
- 触发现有 104/106 切图事务；
- 目标层重定位确认后完成出口段；
- 恢复平地步态并把控制权交还 Nav2。

楼梯执行器不能包含平地路径规划器，不能维护第二份任务队列，也不能绕过当前仲裁直接向 103 写速度。

## 八、共享平台切图与自动重定位

机器狗到达共享平台后先停止，保持 `terrain_guard` 的平台安全确认，不立即恢复 Nav2。切层由 `floor_manager` 作为唯一事务所有者；每次生成持久化 `request_id` 和单调递增 `map_epoch`，状态只允许按以下顺序推进：

```text
PREPARED -> APPLYING -> RELOCALIZING -> COMMITTED
                         |
                         +-> ROLLING_BACK -> ROLLED_BACK | UNCERTAIN
```

服务重启、任务取消或超时后，旧 `request_id/map_epoch` 的迟到线程和结果不得再提交当前楼层。系统必须先恢复未完成事务，不能同时启动第二个切层动作。

正式事务步骤如下：

1. 校验活动任务、路线 ID、源地图、目标地图、走廊版本、内容摘要和平台锚点完全一致；确认机器狗零速、平地姿态，并处于 `source_platform` 的位置和朝向容差内。
2. 对 106 执行目标层 `drmap apply`，随后读取 active 地图身份和摘要做后验确认；SSH 命令退出码不能单独作为成功证据。
3. 对 104 加载目标层 Nav2 地图，按 YAML、图像内容摘要和 `/map` 身份确认目标地图；不能只比较尺寸、分辨率和原点。
4. 使用路线中保存的 `target_platform x/y/z/yaw` 发起本次唯一的 2101 重定位；不得把源地图坐标直接带入目标地图。
5. 以“目标地图确认完成且 2101 已发出”为时间屏障，等待属于当前 `map_epoch` 的 2101 回执、原厂定位状态、目标地图位姿和平台锚点位置/朝向误差通过。
6. 清空两张代价地图，确认 Nav2 lifecycle active、目标 TF 可变换，并等待时间屏障之后新产生的 `/scan`、局部代价地图和全局代价地图全部恢复。
7. 在连续稳定窗口内确认位姿无超限漂移或跳变；单帧位姿和“扫描仍在发布”都不足以证明目标地图定位稳定。
8. 全部通过后持久化 `COMMITTED`，再更新 `current_floor`，允许 `stair_executor` 执行目标层出口段。

跨层自动重定位必须使用独立于普通人工重定位的严格锚点容差，同时校验位置和朝向；不能继续使用当前通用接口的 `2.0m` 位置容差。具体数值由共享平台录包标定后进入唯一现场配置文件，不在代码中硬编码。

当前切层实现距离上述事务还有以下根本缺口，必须整体收口后才能解除跨层门禁：

1. 现有代码以 `relocalization.confirmed` 判断成功，而 `/scan` 和两张代价地图只进入 `navigation_ready`；正式逻辑必须同时要求：

```text
relocalization.confirmed == true
and relocalization.navigation_ready == true
```

2. 当前通用位置容差为 `2.0m`，朝向误差虽然被计算但没有进入成功门限；跨层必须使用独立、统一且经现场标定的位置/朝向容差。
3. 当前 106 apply 主要依赖命令退出码，104 `/map` 主要比较元数据；二者都缺少不可变内容摘要的后验确认。
4. 当前后台切图线程缺少持久事务阶段和迟到结果隔离，超时、取消或进程重启后可能留下无法判定的物理状态。
5. 当前回滚只恢复源地图，没有使用 `source_platform` 重新执行 2101 并验证源层导航恢复。

任一条件失败，都不能报告切层成功。“回切源地图”也不等于回滚完成：系统必须确认 106/104 源地图身份和摘要，使用 `source_platform` 重新执行 2101，并重新通过定位稳定窗口、TF、`/scan` 和两张代价地图检查，才能提交 `ROLLED_BACK`。此前始终保持 `current_floor` 未知、运动禁止和 `state_uncertain=true`；回滚无法闭环时要求人工选择地图和重定位。

## 九、异常处理和安全边界

以下任一条件发生时，楼梯执行器必须停止运动并保持任务失败或等待人工处理：

- 三维地形状态为 `blocked`、`unknown` 或 `stale`；
- 点云覆盖不足或配置版本不一致；
- 当前路线、地图、楼层或走廊版本不一致；
- 横向偏移、航向偏差或姿态超过已验证边界；
- 步态切换没有得到确认；
- 机器狗长时间无有效前进；
- 104/106 任一切图步骤失败；
- 2101 重定位没有新鲜成功证据；
- 用户停止任务、手柄接管或软急停；
- 系统重启后存在未完成的跨层事务。

楼梯上不执行以下自动恢复动作：

- 原地旋转；
- 自动倒退脱困；
- 清空代价地图后继续；
- 绕开人员或障碍继续爬楼；
- 在地图身份不确定时猜测当前楼层。

遇到人员或临时障碍时先停止。障碍离开后是否允许继续，必须重新完成时效、走廊、姿态和路线一致性检查；不能因为上一帧曾经 `traversable` 就自动恢复。

## 十、实施顺序

### 阶段 0：保持当前安全基线

- 完成旧楼梯扫描选择器、旧净空分类器和旧执行入口撤除。
- `/scan` 恢复为平地唯一导航感知源。
- 跨层执行继续以 `stair_execution_retired` 失效保护拒绝。

### 阶段 1：统一路线和地图数据模型

- 把现有四点路线迁移为有向语义连接边。
- 补齐地图版本、走廊版本、上下楼方向和执行策略。
- 保持现有前端、任务和 API 只使用这一套数据源。
- 删除迁移完成后的兼容分支，不能长期并行两种路线模型。

### 阶段 2：离线派生地图

- 使用现有 PCD 或录包验证高程和台阶连续性算法。
- 自动生成楼梯候选走廊和派生导航图。
- 建立原始地图、派生地图、路线和感知配置的版本关系。
- 本阶段不发送任何真实爬楼运动。

### 阶段 3：106 在线地形安全影子运行

- 在 106 部署 `m20pro_terrain_guard`，仅输出状态，不参与控制。
- 分别录制平地、空楼梯、人员遮挡、箱体、缺口、上楼和下楼数据。
- 根据录包确定参数和时效边界，所有参数进入唯一现场配置文件。
- 影子结果未达到稳定标准前不解除执行门禁。

### 阶段 4：楼梯执行器无负载闭环

- 先验证状态机、仲裁、步态切换、停止和任务恢复，不执行真实楼梯。
- 使用回放和模拟状态验证每个失败分支。
- 确保系统重启、网络中断和任务取消后不会遗留运动权限。

### 阶段 5：最小真实楼梯验证

- 只测试一段结构规则、空间充足的固定楼梯。
- 先上楼单程，再下楼单程；下楼必须后退。
- 现场有人看护，原厂手柄急停可用，全程录包。
- 本阶段只验证楼梯段，不同时叠加雷达、YOLO和复杂巡检任务。

### 阶段 6：接入原子切图和跨层任务

- 楼梯段稳定后接入共享平台切图事务。
- 先验证相邻两层单程，再验证往返。
- 最后加入目标层任务点、停留、雷达巡检和多点任务。

## 十一、解除跨层执行门禁的验收条件

只有以下条件全部满足，才能移除 `stair_execution_retired`：

1. 单层导航和动态避障回归通过，楼梯模块关闭时行为与稳定基线一致。
2. 楼梯路线、地图、走廊和配置均有明确版本且能在录包中追溯。
3. 三维地形感知在空楼梯上持续给出正确通行结果。
4. 人员、箱体、点云中断和低覆盖场景均能进入停止状态。
5. 楼梯执行器所有控制输出都经过现有仲裁和看门狗。
6. 上楼、下楼、停止、人工接管和重新开始均完成现场验证。
7. 104/106 切图、2101 重定位、目标层 `/scan` 和两张代价地图恢复证据完整。
8. 任一步骤失败后任务不会越过失败点继续执行。
9. 服务重启后不会把未完成事务误判为已到目标楼层。
10. 现场录包能够完整复盘路线、感知、步态、速度、切图和重定位。
11. 每条有向路线的共享平台完成多次独立重定位和停车重复性验收，锚点与地图摘要一致。
12. 切层事务的超时、取消、服务重启和迟到结果均完成故障注入，不能错误提交目标楼层。
13. 回滚必须恢复源地图、源锚点定位和完整导航 readiness，不能只验证地图文件切回。

## 十二、明确不采用的做法

正式实现禁止以下方向：

- 不整体迁移 Open-RMF 到机器狗；
- 不把 X30Pro 的闭源 ROS1 二进制复制到 M20Pro；
- 不把 Nav2 Route Server 强行回移植为第二套路由系统；
- 不恢复旧 `/m20pro/navigation_scan` 扫描选择链；
- 不用一个高度阈值同时解释台阶和障碍物；
- 不在 104 上长期接收 106 原始点云；
- 不让普通 DWB 在楼梯上尝试自由绕障；
- 不直接覆盖原始地图；
- 不根据 `F1/F2` 名称自动推断连接关系；
- 不在没有厂家机械能力边界的情况下仅靠调高参数测试临时高台阶。

## 十三、最终意见

这条路线最稳妥的原因，不是它代码最少或开发最快，而是它把四类风险分开管理：

- 平地导航风险由现有 Nav2 负责；
- 楼梯运动风险由专用执行器负责；
- 三维地形风险由 106 感知负责；
- 地图身份风险由现有原子切图和重定位事务负责。

每一层只有一个正式责任主体，层与层之间使用版本化、可录包、可验证的接口。任何一层未准备好都不会拖累其他层，也不能带病进入下一阶段。

因此，本项目后续跨楼层开发应以本报告路线为唯一依据。当前失效保护继续保留，先完成语义路线模型和离线派生地图，再建设 106 三维安全感知与专用楼梯执行器；不得再围绕旧扫描链做局部修补。

## 十四、技术依据

- X30Pro 原厂多楼层建图和导航资料：连续三维定位、分层二维地图、共享平台切图、显式步态和方向。
- Nav2 Route Server：有向路线图、语义边和进入/离开边时触发操作。
- Open-RMF：建筑、楼层高度、楼层地图和跨层连接器的数据建模。
- ANYbotics Grid Map、Elevation Mapping、Traversability Estimation：高程、坡度、粗糙度和台阶连续性计算。
- M20Pro 当前实测主线：106 edge scan、104 Nav2、103 TCP、104/106 原子切图及 2101 重定位确认。
