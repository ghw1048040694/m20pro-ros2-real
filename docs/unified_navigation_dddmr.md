# 统一导航与 DDDMR 边界

## 统一任务模型

系统不区分“单层导航引擎”和“跨层导航引擎”。两者共用同一个 `unified_navigation_plan`：

```text
unified_navigation_plan
  floor_segments: [map + ordered waypoints]
  transitions:    [directed stair connector]
```

单层是 `floor_segments` 只有一项、`transitions` 为空的最小输入。跨层只在连续楼层段之间增加有向连接边，不复制任务队列、点位库或地图库。

任务保存点位 ID、连续楼层段、地图 ID 和有向路线 ID。执行时从现有点位库和路线表恢复位姿，不另存第二份几何真值。

## 当前运动边界

```text
平地：106 edge scan -> /scan -> 104 Nav2
楼梯：stair_executor -> 原厂楼梯步态 + 低速直行
切层：现有 104/106 地图事务 + 目标层 2101
```

唯一 `stair_executor` 负责入口 Nav2、步态切换、楼梯直行、平台停车、切图请求、出口 Nav2 和恢复平地步态。已删除 `stair_action_orchestrator`，旧 `floor_manager.use_stairs()` 保持停用，不形成两套运动链。

`stair_executor` 只向现有 `/cmd_vel_nav` 发布楼梯段速度，下游仍由 `command_mux` 处理仲裁和看门狗。上楼正向，下楼后退。楼梯段不使用 DWB 自由规划，入口和出口仍使用现有 Nav2。

地图切换只在共享重叠停车区进行，不在地图边界线上进行。104 和 106 的地图资产在任务前预置，停车后并行激活两端目标地图，再在目标图的 `target_platform` 自动执行 2101。运行时不下载地图、不计算跨主机哈希，也不建立第二套定位器。

`m20pro_real_full.sh shadow` 不启动楼梯运动。`m20pro_real_full.sh move` 随轴运动能力启动楼梯执行器。原始 launch 参数默认仍为 `false`，避免只开执行器却没有 103 运动链。

## DDDMR 的使用方式

DDDMR 只作为后续三维地形感知的设计参考，不把它的 ROS 版本、Docker、定位器或整套导航栈搬入 M20Pro。值得保留的只有三个思路：

1. 静态可通行地形与实时障碍分开。
2. 原始点云就地处理，只向运动链输出轻量结果。
3. 楼层内全局路线、楼梯段执行和局部风险感知分属不同责任边界。

106 上的 `terrain_guard` 保留为可选影子观测：

```text
106 /LIDAR/POINTS -> terrain_guard -> 轻量地形状态
```

它不跨主机传原始点云，不修改 `/scan`，不替换 Nav2 代价地图，首轮也不阻止楼梯执行。等最小闭环跑通且影子数据证明它能稳定识别真实风险后，再评估是否接入停止条件。

## 开发顺序

1. 先跑通两层、一条上行路线的最小真机闭环。
2. 确认入口、步态、平台停车、切图重定位、出口和继续任务的顺序真实可运行。
3. 再测试反向下楼、往返和多连接边任务。
4. 根据实测录包确认真实失效模式，逐项加固。
5. 最后才把 106 地形影子结果接入楼梯停止逻辑。

## 明确不做

- 不把 DDDMR 或其他导航栈整套搬入 104/106。
- 不在 104 长期订阅 106 原始点云。
- 不恢复旧楼梯扫描选择器。
- 不并行运行两个楼梯执行器或两个任务队列。
- 不在首次真机闭环前叠加 terrain 强制门禁、额外路线认证或新仲裁层。
